# Report — AI実行レポートを作る

人間がエージェントへ指示した作業が終わったら、何を見てどう判断し、どこに摩擦があったかまで含めて root `memory_bank/report/` に残す。
これは `/report` と言われた時だけの任意操作ではなく、すべての指示に対する完了条件として扱う。

---

## トリガー

- `/report`
- 「レポート書いて」「report 書いて」「作業報告」
- 人間がエージェントへ指示した作業を完了するとき

---

## 入力

以下のいずれか:

- **対応する plan がある場合** → plan ファイルのパスを指定（または自動検出）
- **plan なしの場合** → ユーザーに「何をやりましたか？」と質問する
- **何も指定なし** → 今回のセッションでの作業内容から自動でまとめる

---

## ワークフロー

### Step 1: 作業内容の収集

以下の方法で作業内容を把握する（優先度順）:

1. 対応する plan がある → plan の「やること」をベースに差分を確認
2. ユーザーが説明する → その内容をもとに整理
3. セッション内の作業から推測 → 変更したファイル、実行したコマンド等から自動まとめ

**判断基準**: 「次のセッションでこの report を読んだとき、非エンジニアでも30秒で全体像が把握できるか？」

### Step 2: 変更内容の整理

- 変更したファイルの一覧を作成する
- 各変更の概要を1行で書く
- 判断の理由を記載する（なぜこの方法にしたか）
- `used_rules`、`used_workflows`、`used_skills`、`used_knowledge`、`primary_issue_layer`、`friction_score` を埋める

**判断基準**: 「理由なき変更がないか？ すべての変更に "なぜ" が説明できるか？」

### Step 3: 残タスクと学びの記録

- 未完了のタスクがあればリストアップする
- 作業中に気づいたこと・次回に活かせる学びを書く
- plan がある場合、完了条件との照合結果を記載する
- `まずこれだけ見ればOK`、`AIが見たもの`、`次回もっと良くするには` の 3 セクションを必ず入れる
- 診断ログとして `blocking_factors`、`root_cause`、`quality assessment`、`suggested_fix` を残す

---

## 出力フォーマット

`memory_bank/report/_TEMPLATE.md` に合わせる。

---

## 保存先

`memory_bank/report/YYYY-MM-DD_{タイトルスラッグ}.md`

---

## セルフチェック

1. 次回のセッションでこの report を読んだとき、30秒で全体像がわかるか？
2. 「何を変えたか」だけでなく「なぜ変えたか」が書かれているか？
3. `AIが見たもの` と診断ログの両方があり、管理側の分析にも使えるか？
