# スキル監査

スキルファイル（SKILL.md）の品質・整合性・鮮度を構造的にチェックし、具体的な改善提案を出す。

---

## トリガー

- `/skill-audit` — 全スキルを監査
- `/skill-audit [スキル名]` — 指定スキルを監査
- 「スキル監査」「スキルチェック」「スキルの状態」

## 入力

- 引数なし → 全スキルを監査
- `{skill_name}` → 単一スキル

---

## ワークフロー

### Step 1: 対象スキルの特定

1. `.agents/skills/` 以下のディレクトリをGlobで列挙
2. 各ディレクトリ内の `SKILL.md` を確認
3. 引数指定があればそのスキルのみ

### Step 2: 各スキルの監査（8項目）

対象スキルごとに以下を実行:

1. **SKILL.md を Read**
2. 依存するルール/ナレッジがあれば確認

#### 8項目の健全性チェック（各12.5点 = 100点満点）

| # | 項目 | チェック内容 |
|---|------|------------|
| 1 | **構造の完全性** | トリガー・入力・ワークフロー・出力フォーマット・セルフチェックが揃っているか |
| 2 | **指示の明確さ** | ステップバイステップで記述されているか。曖昧な「適切に」「必要に応じて」が多用されていないか |
| 3 | **判断軸の有無** | 各ステップに「何を基準に判断するか」が埋め込まれているか（2層構造の抽象層） |
| 4 | **依存パスの正確性** | SKILL.md内で参照しているファイルパス・ツール名が実在するか |
| 5 | **Frontmatter完全性** | name, description, version が正確に記載されているか |
| 6 | **出力フォーマットの具体性** | テンプレートが書かれているか。「いい感じに」で終わっていないか |
| 7 | **セルフチェックの質** | チェック項目が具体的で検証可能か |
| 8 | **再利用性** | テーマを変えても使えるか。特定トピック専用になっていないか |

#### 採点基準

| 点数 | 基準 |
|------|------|
| 12.5 | 完全に健全。問題なし |
| 9 | 軽微な改善余地あり |
| 6 | 明確な問題が1つ以上 |
| 3 | 重大な問題がある |
| 0 | 完全に破綻/未実装 |

### Step 3: 修正提案の生成

問題が見つかった項目ごとに、修正案を出す:

| 分類 | 説明 |
|------|------|
| **AUTO-FIX** | 自動修正可能（パス修正、frontmatter補完等）→ ユーザー確認後に修正 |
| **DRAFT-FIX** | 修正案を生成 → diff形式で表示 → ユーザー確認後に適用 |
| **HUMAN-REVIEW** | 判断が必要 → 問題点と選択肢を提示 |

### Step 4: レポート出力

```markdown
# スキル監査レポート（{YYYY-MM-DD}）

## サマリー
- 監査対象: {N}スキル
- 健全（90+）: {N}スキル
- 要注意（70-89）: {N}スキル
- 要即対応（70未満）: {N}スキル

## 詳細

### {skill_name}（{score}/100 → {健全/要注意/要即対応}）

| # | 項目 | 点数 | 状態 | 詳細 |
|---|------|:----:|------|------|
| 1 | 構造の完全性 | {/12.5} | {OK/WARN/NG} | {詳細} |
| ... | ... | ... | ... | ... |

#### 修正提案
- [AUTO-FIX] {修正内容}
- [DRAFT-FIX] {修正案の概要}
- [HUMAN-REVIEW] {判断が必要な事項}

---
```
