### 読み上げ原稿

#### スライド1: セクション9タイトル
セクション9では「議事録からタスク一覧を作る」に取り組みます。このセクションでは、議事録をタスク一覧と担当者別連絡文に変換する仕組みを作ります。完成すると、会議後10分の手作業が2分になります。キーワードは「曖昧をつぶす」です。

1時間の会議が終わった。議事録には「検討します」が5回。2週間後、誰も何もしていない。この"消える決定事項"、経験ありませんか？ AIが「検討します」を見つけて、「いつまでに？誰が？」という確認質問を自動で作ってくれます。

会議が終わった後の作業を考えてみてください。議事録を読み返して、誰が何をやるか整理して、メールやSlackで担当者に連絡する。この一連の手作業に、どのくらい時間を使っていますか？

このセクションでは、Claude Codeを使って議事録から「決定事項・ToDo・論点」を自動で分類し、タスク一覧まで作る方法を学びます。

#### スライド2: 「そのまま置かない」を徹底する
議事録を「そのまま保存しても後で使えない」という話から始めます。

会議のメモやテキストをそのまま保存しても、後で「あの会議で何が決まったっけ」と探す羽目になります。原因は「整理されていないから」です。

解決策はシンプルです。3つに分けてから保存します。

1つ目が「決定事項」。会議で確実に決まったことです。

2つ目が「ToDo」。誰かがやるべきことです。

3つ目が「論点」。まだ決まっていないことです。

この3分類に入れるだけで、後の作業が大きく変わります。

#### スライド3: 議事録テンプレートの構造
テンプレートを見てみましょう。

```markdown
# 議事録：[会議名]（[日付]）

## 決定事項
- トップページのリニューアルはA案で進行する

## ToDo
- 企画書の初稿作成（担当：田中、期限：3/10）
- 費用見積もり依頼（担当：鈴木、期限：3/8）

## 論点（未決定）
- 価格設定：次回MTGで最終決定予定

## 次回MTG
- 日時：3月15日（土）14:00
```

「決定・ToDo・論点・次回MTG」の4セクション構成です。どんな会議にも使えます。

特に重要なのが「担当者と期限を必ず書く」というルールです。担当者が不明なToDoは「担当：未定」と明記します。

### 読み上げ原稿

#### スライド1: レクチャー導入
このレクチャーが「セクション9の核心」です。

チェック（品質保証）の深掘りとして、「曖昧表現の検出」をやります。少し時間をとりますが、ここをしっかり学ぶと議事録の使い方が大きく変わります。

#### スライド2: 曖昧表現とは何か
「曖昧表現」というのは、会議でよく出てくる言葉で、後から「言った・言わない」の原因になるものです。

例を見てみましょう。

「検討します」。これを聞いて「決まった」と思う人と「まだ決まっていない」と思う人がいます。いつまでに、誰が、何を判断するのか、一切書いていません。

「共有します」。何を？どの手段で？いつまでに？これも不明です。

「できれば」。やるのか？やらないのか？条件は何か？

「前向きに」。具体的なアクションは何か？

「確認しておきます」。何を？誰に？いつまでに？

これらが議事録に残ったまま会議が終わると、次のMTGで「あれどうなりましたか？」が起きます。

#### スライド3: 曖昧表現辞書
配布資料に「曖昧表現辞書」を入れています。これを見てください。

```
| 曖昧表現 | 危険度 | 確認質問 | 変換例 |
| 検討します | 高 | いつまでに？誰が？結果報告先は？ | 「3/5までに田中が検討し、Slackで報告」 |
| 共有します | 中 | 何を？どの手段で？いつまでに？ | 「議事録を3/2中にSlack #generalに投稿」 |
| できれば | 高 | やるのか？やらないのか？条件は？ | 「予算が承認されたら実施（3/10判断）」 |
| 前向きに | 高 | 具体的な次アクションは？ | 「3/5に見積もりを取得し、3/7に判断」 |
| なるべく早く | 高 | 具体的な期限は？ | 「3/4 17:00まで」 |
| 後日 | 高 | いつ？ | 「来週月曜（3/8）まで」 |
```

「危険度」という列があります。「高」は期限・担当者・完了条件のどれも不明なもので、放置するとタスクが消滅するリスクがあります。

この辞書はカスタマイズできます。自分の職場でよく出る表現を後から追加できます。

#### スライド4: 曖昧表現を「見つける理由」
ここで重要なことをお伝えします。

曖昧表現の検出は「全部をなくすのが目的ではありません」。

見つけて「確認する機会を作る」ことが目的です。会議中に全部明確にするのは現実的に難しい。だから会議後のチェックで補います。

これが議事録チェックの一番大事なポイントです。「言った・言わない」を防ぐための、事後的な確認ステップです。

#### スライド5: Claude Codeへの指示例
では指示の内容を確認します。

```
以下の議事録テキストを読んで、
曖昧な言い方が含まれる部分を抜き出してください

対象となる曖昧表現：
「検討します」「共有します」「できれば」「前向きに」
「確認します」「調整します」「なるべく早く」「後日」

各曖昧表現に対して：
1. 該当箇所を引用
2. 確認質問を1つ生成
3. 確認が必要な担当者名（わかれば）

出力形式：テーブル（発言者 / 発言 / 危険度 / 確認質問）
```

「対象となる曖昧表現のリストは自分で増やせる」という点を覚えておいてください。自分の職場に合わせてカスタマイズできます。

#### スライド1: ToDoとタスクは違う
「ToDo」と「タスク」は別物です。

ToDoは「やること」だけ書いてあります。「企画書を作る」これだけです。

タスクは「誰が・いつまでに・何をするか」が揃っています。「田中さんが3月10日までに企画書の初稿をWordで作り、GoogleDriveに共有する」これがタスクです。

#### スライド2: tasks.csvのフォーマット
タスク一覧をCSVで出力します。列の構成を見てください。

```
id,task,assigned_to,due_date,status,next_action

2,費用見積もり依頼,鈴木,2026-03-08,未着手,ベンダーにメール送信
```
