#### スライド1: セクション5タイトル
セクション5は「安全に使う」というテーマです。このセクションでは、止める・戻す・守る。この3つの安全装置を入れます。完成すると、「何かあっても大丈夫」という安心感を持って作業できるようになります。

「試してみたいけど、なんか怖い」と感じている方、まさにそのための時間です。怖さの正体のほとんどは「何かやらかしたときに取り返せるか分からない」という不安です。対処法を先に知っておけば、その不安はかなり減ります。

#### スライド2: 5-1レクチャータイトル
このレクチャーで覚えることは3つです。止める、巻き戻す、そしてGitで復元する、の3つです。この3つを頭に入れておくだけで、ほとんどのトラブルに対応できます。

#### スライド3: このレクチャーで達成すること

3つを整理します。

1つ目が「止める」。Claude Codeが動いている最中に、あ、これは違う、と気づいたときにすぐ止める方法です。キーボードのCtrl+Cで即停止できます。MacでもWindowsでも同じです。

2つ目が「巻き戻す」。直前の変更をなかったことにしたいときに使います。`/rewind` というコマンドを入力するか、Escキーを2回押すと、会話とファイルの変更をまとめて戻せます。

3つ目が「Gitで復元する」。これは数日前の状態に戻したいときに使います。少し難しそうに見えますが、ゲームのセーブポイントと同じだと思ってください。

## レクチャー 5-2: 確認が多すぎる時の設定（想定時間: 6分）

### 画面構成
- 使用画面: スライド（メイン）→ Claude Code設定画面（画面共有）→ スライド
- 切替タイミング: 「おすすめの設定」スライドで実際の設定ファイルを表示

### 読み上げ原稿

#### スライド1: 5-2レクチャータイトル
Claude Codeを使い始めると、最初のうちは「本当に実行しますか？」という確認が頻繁に来ます。これは安全のためですが、慣れてくると毎回の確認が少し面倒に感じることもあります。

このレクチャーでは、確認の頻度を調整する方法を学びます。ただし、全部OKにしてしまうのはお勧めしません。そのバランスをお伝えします。

#### スライド2: このレクチャーで達成すること
1つ目は権限モードの種類と使い分けを理解すること。2つ目は初心者に推奨の設定を確認すること。3つ目は全自動承認にしない理由を理解することです。

#### スライド3: 権限モードの3種類
権限モードは3種類あります。

1つ目が Default（通常）モード。ファイル変更のたびに確認が来ます。最初はこれを使ってください。

2つ目が Plan（計画確認）モード。実行前に「こういう手順でやります」という計画を見せてくれます。慣れてきたらこちらがおすすめです。

3つ目が Auto-Accept Edits（編集の自動承認）モード。ファイルの編集は自動で承認されますが、コマンド実行などは引き続き確認が来ます。完全な全自動ではないのがポイントです。

車の運転モードに例えると、Defaultは教習所です。教官が「ここでブレーキ踏んで」と都度確認してくれます。Planは一般道です。ルートだけ先に確認して、あとは自分で運転します。Auto-Acceptは高速道路です。スピードは出ますが、事故のリスクも上がります。

おすすめは最初は Default、慣れてきたら Plan です。

#### スライド4: おすすめの設定（初心者向け）

> **[画面切替: 設定ファイル表示]**

設定ファイルで確認してみましょう。実際の設定例はこちらです。

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(rm *)",
      "Bash(git push --force*)"
    ]
  }
}
```

この設定の意味を読み解きます。`allow` に書いたツール、つまりファイルの読み書き・編集は自動で許可されます。`deny` に書いた危険なコマンド、一括削除や強制pushは拒否されます。それ以外は都度確認が来ます。

この `permissions` の `allow` と `deny` でツールごとに細かく制御できます。

> **[画面切替: スライド]**

#### スライド5: 全自動承認にしない理由
`--dangerously-skip-permissions` というオプションがあります。名前から分かる通り、「危険を承知でスキップする」という意味です。

これを使うとファイルの削除も確認なしで実行されます。インターネットへの接続も確認なしになります。間違いに気づくのが遅れます。

このオプションは存在はしますが、使わないことを強くお勧めします。特に初心者のうちは絶対に使わないでください。

#### スライド6: まとめ + 次へ
まとめます。最初は Default モードから始めます。慣れたら Plan モードに変えます。Auto-Accept Edits は慣れてから使いましょう。

次のレクチャーでは、巻き戻せない操作を先に知っておきます。「戻せると思ったら戻せなかった」という事故を防ぐための知識です。

### このレクチャーのポイント
- 「承認疲れ」の共感から入る。毎回確認が来る体験を持っている受講者に刺さる
- Auto-acceptの危険性は具体例（ファイル削除）で伝える
- 設定ファイルは画面で実際に見せると信頼度が上がる
- つまずきやすい点: 設定ファイルの場所が分からない受講者が多い。「Claude Codeの設定画面から開ける」と補足する

## レクチャー 5-3: 巻き戻しの注意点（想定時間: 5分）

### 画面構成
- 使用画面: スライド（全編）
- 切替タイミング: 対比表は画面共有のスライドで十分見える

### 読み上げ原稿

#### スライド1: 5-3レクチャータイトル
`/rewind` は非常に便利なコマンドですが、万能ではありません。戻せない操作があります。これを知らないと「戻したはずなのに変わっていない」という混乱が起きます。

#### スライド2: このレクチャーで達成すること
1つ目は巻き戻せない操作の種類を知ること。2つ目は危険な操作を避ける判断基準を持つこと。3つ目は「戻せる/戻せない」の見分け方を覚えることです。

#### スライド3: 巻き戻せる vs 戻せない（対比表）
整理します。

戻せる操作はこちらです。ファイルの作成、ファイルの編集、フォルダの作成、設定ファイルの変更、Claude Codeの会話履歴。これらはすべてローカルの変更なので、`/rewind` で戻せます。

戻せない操作はこちらです。git push（リモートへの送信）、外部APIへのリクエスト送信、メールや通知の送信、一部のコマンド実行です。

シンプルに言うと「外に出たデータは戻せない」です。この一言を覚えておいてください。

#### スライド4: 特に注意が必要な操作
2つ特に注意が必要な操作を説明します。

1つ目が `git push` です。ローカルの変更をリモートに送信する操作で、一度送ると相手側に残ります。対策は大きな作業の前にブランチを分けることです。ただ、これは少し高度なので、今は「git push は慎重にする」とだけ覚えてください。

2つ目が外部APIの送信です。Slack通知、メール送信などがこれに当たります。送ったら取り消せません。対策はテスト環境で確認してから本番で実行することです。

基本ルールは「外に出す前に確認する」です。

#### スライド5: まとめ + 次へ
まとめます。ファイル変更は戻せます。外部への送信は戻せません。「外に出す前に確認する」が基本ルールです。

次のレクチャーでは、Sandboxという安全な作業場を作る方法を学びます。

### このレクチャーのポイント
- 対比表を声で読み上げるときはゆっくり丁寧に。受講者がメモを取る時間を作る
- 「外に出たデータは戻せない」の一言を繰り返して印象づける
- git pushの説明は深追いしない。「慎重にする」だけ伝えれば十分
- つまずきやすい点: APIの概念が分からない受講者がいる。「Claude CodeがSlackやメールに送信する機能を使ったとき」と具体例で補う

Sandboxを使うと、今の作業フォルダとそのサブフォルダだけが読み書き可能になり、それ以外のファイルは読み取り専用になります。OSレベルでファイルシステムとネットワークのアクセスを制限する仕組みです。

子供が絵の具で遊ぶとき、テーブルに新聞紙を敷きますよね。Sandboxはそれと同じです。汚れても新聞紙を捨てるだけで、テーブルは無傷です。


Sandboxで試すといいことがあります。新しいスキルのテスト、大きなファイル変更の試行、慣れていない操作の練習などです。

Sandbox内で動かしてみて、うまくいったら通常モードに戻して本番で作業する、という使い方がおすすめです。
