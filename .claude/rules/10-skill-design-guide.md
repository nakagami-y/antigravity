# スキル設計ガイド（スキルを作る・改修するときに参照）

スキル（`.claude/skills/*/SKILL.md`）を作るときの設計原則。
「ただのToDoリスト」ではなく「判断力ごと再利用できるスキル」を作るためのガイド。

---

## 核心: 2層構造

すべてのスキルは **具体層** と **抽象層** の2層で設計する。

| 層 | 役割 | 欠けるとどうなるか |
|----|------|------------------|
| **具体層**（手順） | 何をやるか、どの順番でやるか | AIが動けない。出力がブレる |
| **抽象層**（判断軸） | どの視点で見るか、何を優先するか | ただのToDoリスト。状況が変わると対応できない |

### 良いスキルの構造

```
SKILL.md
├── トリガー / 入力          ← 具体層（いつ・何を渡して起動するか）
├── 判断フレーム              ← 抽象層（どう考えるか）
├── ワークフロー（Step 1-N）  ← 具体層 + 各Stepに判断軸を埋め込む
├── 出力フォーマット          ← 具体層（テンプレート）
└── セルフチェック            ← 抽象層（品質の判断基準）
```

---

## 具体層の必須要素

1. **トリガー**: `/スキル名` + 自然言語パターン（「○○して」「○○作って」）
2. **入力定義**: 必須/任意を明記。型と例を書く
3. **ワークフロー**: Step N 形式。依存関係がある場合は順序を明示
4. **出力フォーマット**: テンプレートを書く。「いい感じに」は禁止
5. **保存先**: ファイルを作る場合はパスを明示

---

## 抽象層の埋め込み方（3パターン）

### パターンA: 分岐ロジック

入力の状態によってワークフローが変わる。

```markdown
### Step 2: 判定 → 分岐

| 入力の状態 | 判定 | 次のアクション |
|-----------|------|-------------|
| 完成した記事 | そのまま使える | Step 3 へ |
| メモ・箇条書き | 構造化が必要 | Step 2B で整理 |
| 情報不足 | 書けない | ユーザーに追加情報を聞く |
```

### パターンB: 評価基準

生成物の良し悪しを「感覚」ではなく「基準」で判断する。

```markdown
### セルフチェック

- 対象者がこれを読んで「次も読みたい」と思うか？
- 読者が「今日これをやる」と決められるか？
- 同テーマの他の記事と並べたとき、選ばれる理由があるか？
```

### パターンC: メタ認知（自分の出力を疑う）

AIは自分の出力を疑わない。だからスキルに「疑え」という指示を埋め込む。

```markdown
### 最終チェック

1つでも No なら修正:
1. この出力を自分の名前で公開して恥ずかしくないか？
2. 読者は「で、何すればいいの？」と思わないか？
3. 3日後に読み返して「AIが書いたな」と思わないか？
```

## ダメなスキルの見分け方

| アンチパターン | 症状 | 対処 |
|--------------|------|------|
| **ToDoリスト症候群** | 全Stepが「○○する」だけで判断基準がない | 各Stepに「何をもってOKとするか」を追記 |
| **抽象論オンリー** | 「良い記事を書く」のような曖昧な指示 | 具体的な基準・例・テンプレートを追加 |
| **巨大モノリス** | 10以上のStepが1ファイルに | 判断/生成/評価で分割を検討 |

## スキル作成チェックリスト

- [ ] トリガー（`/スキル名`）が決まっているか
- [ ] 入力が明確か（何を渡すとスキルが動くか）
- [ ] 各Stepに判断基準があるか（「やる」だけでなく「何をもってOKか」）
- [ ] 出力フォーマットがテンプレートで定義されているか
- [ ] セルフチェック（最低3つの問い）があるか
- [ ] 分岐ロジックがあるか（入力の状態で動きが変わるか）
