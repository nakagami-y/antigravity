# Setup Wizard — Claude Code セットアップウィザード

初めて Claude Code を使う人が、対話に答えていくだけで自分専用の設定を構築できるスキル。

---

## トリガー

- `/setup-wizard`
- 「セットアップして」「初期設定して」「設定ウィザード」

---

## ワークフロー

### 全体の流れ

5つのレベルを順番にクリアしていく。
各レベルでユーザーに質問し、回答をもとにファイルを生成する。
**1レベルごとに「完了！次のレベルに進む？」と確認する**（一気に進めない）。

開始時に以下のメッセージを表示:

```
============================================
  Claude Code セットアップウィザード
============================================

5つのレベルをクリアして、あなた専用のAI環境を作ります。

  Lv.1  プロジェクトの自己紹介     → CLAUDE.md
  Lv.2  AIのキャラを決める        → rules/02-style.md
  Lv.3  最初のスキルを作る         → skills/{名前}/SKILL.md
  Lv.4  作業ログの仕組みを知る     → memory_bank/ の使い方
  Lv.5  仕上げチェック            → 全体の整合性確認

  所要時間: 10〜15分
  いつでも「一旦やめる」で中断OK

============================================

では、Lv.1 から始めましょう！
```

---

### Lv.1: プロジェクトの自己紹介（→ CLAUDE.md を更新）

**目的**: CLAUDE.md にプロジェクトの基本情報を書き込む。

**ユーザーへの質問**（AskUserQuestion で1問ずつ聞く）:

1. 「このプロジェクト（フォルダ）は何のためのものですか？」
   - 選択肢: Webアプリ開発 / コンテンツ制作（ブログ・SNS等） / 業務自動化 / 学習・実験用 / その他

2. 「一言で表すと、何を作っている（やっている）プロジェクトですか？」
   - 自由入力

3. 「誰のために作っていますか？（対象ユーザー/読者）」
   - 自由入力

4. （開発プロジェクトの場合のみ）「使っている言語・フレームワークは？」
   - 自由入力

**生成アクション**:
- 回答をもとに `CLAUDE.md` の「プロジェクト概要」セクションを Edit で更新する
- 開発プロジェクトなら「コードスタイル」セクションも埋める

**完了メッセージ**:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━
  Lv.1 クリア! CLAUDE.md を更新しました
━━━━━━━━━━━━━━━━━━━━━━━━━━

保存先: ./CLAUDE.md

これで Claude Code はあなたのプロジェクトを理解できるようになりました。
次回から、Claude はこのファイルを毎回自動で読みます。

次は Lv.2「AIのキャラを決める」です。進みますか？
```

---

### Lv.2: AIのキャラを決める（→ rules/02-style.md を生成）

**目的**: Claude の応答スタイル（キャラ・口調）を設定するルールファイルを作る。
ここが一番楽しいところ。好きなキャラで設定すると、AIとの会話が一気に楽しくなる。

**導入メッセージ**:
```
Lv.2: AIのキャラを決める

Claude Code は口調やキャラクターを自由に設定できます。
好きなキャラの話し方にすると、AIとの会話が格段に楽しくなります。

例:
- 「優しい先輩」風 → 丁寧だけど堅くない、質問しやすい
- 「鬼コーチ」風 → 厳しいけど的確、妥協を許さない
- 「友達」風 → タメ口、雑に相談できる
- 好きなキャラクター風 → 口癖や語尾を指定して再現
```

**ユーザーへの質問**:

1. 「Claude にどんなキャラで話してほしいですか？」
   - 選択肢:
     - ビジネスパートナー（丁寧・フォーマル・プロフェッショナル）
     - カジュアルな先輩（です・ますだけど堅くない・質問しやすい）
     - 友達（タメ口・フランク・気軽に相談できる）
     - 好きなキャラクターを指定する（自由入力）

2. （「好きなキャラクター」を選んだ場合）「どんなキャラですか？口癖や話し方の特徴を教えてください」
   - 自由入力（例: 「〜だぜ、が口癖」「ツンデレ」「語尾に"にゃ"がつく」等）

3. 「回答の長さの好みは？」
   - 選択肢:
     - 簡潔重視（要点だけ。説明は聞かれたら）
     - バランス型（要点＋必要な補足）
     - 丁寧重視（背景説明や理由も含める）

4. 「その他、こだわりがあれば教えてください（なければスキップ）」
   - 自由入力（例: 「絵文字使わないで」「箇条書き多めで」「英語のまま表示して」等）

**生成アクション**:
- 回答をもとに `.claude/rules/02-style.md` を Write で生成する
- キャラクター指定の場合は、口調ルール・語尾・一人称・禁止表現を構造化して書く
- テンプレート:

```markdown
# 出力スタイル（全会話に自動適用）

## キャラクター設定

{選択に応じたキャラクター定義を記述}

## 基本の口調

{選択に応じた口調ルールを記述}

## 回答の長さ

{選択に応じた長さルールを記述}

## その他のルール

{自由入力の内容を反映}

## 注意

- 技術的な説明の正確さはキャラに関係なく維持する
- コードブロック内はキャラ口調にしない（コードは正確に）
- ファイルパスや技術用語はそのまま正確に表記する
```

**完了メッセージ**:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━
  Lv.2 クリア! キャラ設定を作成しました
━━━━━━━━━━━━━━━━━━━━━━━━━━

保存先: .claude/rules/02-style.md

次から Claude はこのキャラで話します。
気に入らなければ、いつでもこのファイルを編集するだけでOK。

次は Lv.3「最初のスキルを作る」です。進みますか？
```

### Lv.3: 最初のスキルを作る（→ skills/{名前}/SKILL.md を生成）

**目的**: ユーザーの「よくやる作業」を1つスキル化して、スキルの仕組みを体験させる。

**ユーザーへの質問**:

1. 「Claude にやらせたい"よくある作業"を1つ教えてください」
   - 選択肢（例を見せてイメージさせる）:
     - 日報・議事録の作成
     - コードレビュー
     - SNS投稿の下書き
     - メールの返信案を作る
   - その他（自由入力）

2. 「その作業の入力は何ですか？（何を渡すと作業が始まる？）」
   - 自由入力（例: 「今日やったことのメモ」「レビュー対象のファイルパス」）

3. 「出力はどんな形がいいですか？」
   - 選択肢:
     - Markdownファイルに保存
     - 画面に表示するだけ
     - 両方

**生成アクション**:
- 回答をもとにスキル名（英語スラッグ）を自動生成する
- `.claude/skills/{スキル名}/SKILL.md` を Write で生成する
- スキルの構造:
  - トリガー（`/スキル名` + 自然言語パターン）
  - 入力定義
  - ワークフロー（3ステップ以内でシンプルに）
  - 出力フォーマット
  - 保存先（該当する場合）
  - セルフチェック（3つの問い）

**重要**: 初心者の最初のスキルなので、**シンプルに保つ**。ただし `rules/10-skill-design-guide.md` の最低限（トリガー/入力/ステップ/出力/セルフチェック）は含める。

**完了メッセージ**:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━
  Lv.3 クリア! 初めてのスキルを作成しました
━━━━━━━━━━━━━━━━━━━━━━━━━━

保存先: .claude/skills/{スキル名}/SKILL.md

これで /{スキル名} と入力するだけでこの作業を実行できます。
スキルは何個でも追加できます。「よくやる作業」を見つけたら、
どんどんスキル化していきましょう。

次は Lv.4「作業ログの仕組みを知る」です。進みますか？
```

### Lv.4: 作業ログの仕組みを知る（→ memory_bank/ の使い方）

**目的**: 「AIは過去の作業を覚えられない」という前提を理解させ、作業ログの仕組みを体験させる。

**導入メッセージ**:
```
Lv.4: 作業ログの仕組みを知る

AIには重要な弱点があります:
セッション（チャット）をまたぐと、前に何をやったかを忘れます。

だから「作業ログ」を残します。
AIの記憶の代わりになる、人間とAIの共通の土台です。

このプロジェクトには memory_bank/ フォルダが用意してあります:

  memory_bank/
  ├── plan/     ← 計画書（やる前に書く）
  └── report/   ← 実装レポート（やった後に書く）

  [ plan ] ── 承認 ──> [ 実行 ] ──> [ report ]
    やる前                            やった後
```

**ユーザーへの質問**:

1. 「作業ログの使い方を試してみますか？」
   - 選択肢:
     - やってみる（サンプルの plan を1つ作る）
     - 説明だけ聞いて次に進む

**「やってみる」を選んだ場合**:

1. 「では、今やりたい作業を1つ教えてください（簡単なものでOK）」
   - 自由入力

2. ユーザーの回答をもとに、サンプルの plan を生成して `memory_bank/plan/` に保存する

3. 保存後に説明:
```
これが plan（計画書）です。
実際の作業はこの plan を確認してから始めます。

作業が終わったら report（実装レポート）を残します。
Claude に「report 書いて」と言えば自動で作ってくれます。
```

**追加の説明（どちらの場合も表示）**:
```
もう1つ便利な機能: auto memory

会話中に「これ覚えておいて」と言うと、
次のセッションでも Claude が覚えています。

例:
- 「一人称は"僕"で統一して」→ 覚えておいて
- 「テストは jest を使って」→ 覚えておいて

作業ログ（memory_bank/）と auto memory の使い分け:
- memory_bank/ → 個別の作業の記録（plan / report）
- auto memory  → プロジェクト横断のルール・好み
```

**完了メッセージ**:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━
  Lv.4 クリア! 作業ログの仕組みを覚えました
━━━━━━━━━━━━━━━━━━━━━━━━━━

作業ログのコツ:
- 大きめのタスクの前に plan を書く
- 作業が終わったら report を残す
- 「覚えておいて」で auto memory にルールを保存

最後の Lv.5 で仕上げチェックをします。進みますか？
```

### Lv.5: 仕上げチェック（→ 全体の整合性確認）

**目的**: 作成したファイルの一覧を見せ、全体が正しく構成されているか確認する。

**実行内容**:

1. プロジェクト内の Claude Code 関連ファイルを一覧表示する
2. 各ファイルの1行サマリーを表示する
3. CLAUDE.md を読んでプロジェクトの説明ができるか確認する
4. memory_bank/ の存在を確認する
5. セッション分離のTipsを表示する

**完了メッセージ**:
```
============================================
  全レベルクリア! セットアップ完了!
============================================

あなたの Claude Code 環境:

  CLAUDE.md ..................... プロジェクト指示書
  .claude/rules/01-safety.md ... 安全ルール
  .claude/rules/02-style.md .... キャラ・口調ルール
  .claude/rules/03-tool-tips.md  ツールの使い方と制限
  .claude/rules/04-memory-routing.md  メモリの整理ルール
  .claude/rules/05-quality-check.md   セルフチェック
  .claude/rules/10-skill-design-guide.md  スキル設計ガイド
  .claude/skills/{名前}/ ....... 初めてのスキル
  memory_bank/plan/ ............ 計画書の保存先
  memory_bank/report/ .......... 実装レポートの保存先
  settings.local.json .......... 権限設定

すぐ使えるコマンド（19個）:

  基本ツール:
  /plan   ............... 作業前の計画書を作成
  /report ............... 作業後のレポートを作成
  /review ............... コードや文章をレビュー
  /daily-log ............ 日次の作業ログ（勤務報告）
  /todo ................. タスク管理（追加/一覧/完了）
  /fix-issue ............ イシューの調査→修正ガイド

  コンテンツ制作:
  /brainstorm ........... 8フレームワークでアイデア出し
  /hook ................. フック案を10個生成（H1〜H8型）
  /content-extraction ... 長文→マルチ媒体ネタ抽出
  /reply-draft .......... 返信案を3パターン生成

  リサーチ・分析:
  /competitive-watch .... 競合ウォッチ（4軸スキャン）
  /customer-interview ... ペルソナ対話で顧客理解
  /seo-optimize ......... 一括SEO最適化

  プレゼン・スライド:
  /presentation-builder . スライド構成を設計
  /create-pptx .......... PowerPoint自動生成

  ブラウザ自動化:
  /playwright-cli ....... ブラウザ操作の自動化

  運用・改善:
  /coach ................ 週次1on1振り返り
  /skill-audit .......... スキルの健全性を監査

============================================

Tips: セッションを使い分けよう

  壁打ち・相談 → 考えるためのセッション（新しいチャット）
  実作業       → 正確に進めるためのセッション（別のチャット）

============================================

次のステップ（やりたくなったときに）:

  1. スキルを増やす
     → よくやる作業を見つけたら .claude/skills/ にフォルダを作る
     → 10-skill-design-guide.md を参考にすると質が上がる

  2. ルールを増やす
     → 「毎回こうしてほしい」が見つかったら .claude/rules/ にファイルを追加
     → ファイル名の番号順に読み込まれる（01, 02, 03...）

  3. エージェントを作る（上級）
     → .claude/agents/ にエージェント定義を作ると、
       Task ツールで専門家チームを呼び出せる

  4. メモリを育てる
     → 会話中に「覚えておいて」と言うだけ。どんどん賢くなる

  5. 作業ログを習慣にする
     → /plan で計画書、/report でレポートを自動生成
     → 「plan → 承認 → 作業 → report」のサイクルを回す

  6. レビューを活用する
     → /review でコード・文章・設定ファイルを多角的にチェック

============================================

何か質問があれば聞いてください!
```

## 設計の判断軸

### このウィザードの目的は「完璧な設定を作る」ことではない

**目的は「仕組みを理解させる」こと。楽しみながら。**

- CLAUDE.md が何をするファイルか理解させる
- rules/ が自動適用されることを体験させる（キャラ設定で楽しく）
- skills/ でコマンドが作れることを体験させる
- 作業ログ（plan/report）の習慣を付けさせる
- memory が会話を跨ぐことを知らせる

だから:
- 各レベルで生成するファイルは**最小限**にする
- 複雑なルール（品質ループ、チーム運用等）は入れない
- 「後から足せる」ことを強調する

### ユーザーが「一旦やめる」と言ったら

- 現在のレベルまでの成果を保存して終了する
- 「続きは `/setup-wizard` でいつでも再開できます」と伝える
- 未完了レベルの一覧を表示する

### ユーザーが「わからない」「スキップ」と言ったら

- そのレベルをスキップしてデフォルト値を使う
- 「後からいつでも変更できます」と伝える
- スキップした項目は完了メッセージでリマインドする
