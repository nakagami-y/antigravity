#### スライド4: Skillsがないとき vs あるとき（対比）
まず「Skillsがない状態」を想像してください。

毎月SNSカレンダーを作るとき、「今月のテーマはXXで...」「ターゲットは30代で...」「CSV形式で...」「チェックは...」と毎回同じ指示を長々と書いていませんか？先月うまくいったあの指示、今月また一から書き直していませんか？

これが「Skillsがある状態」になるとどうなるか。毎月たった3行です。

```
/sns-calendar
テーマ: AI活用
期間: 2026年5月
```

これだけで動きます。先月のノウハウがそのままコマンドに入っています。

自動販売機をイメージしてください。お金（入力）を入れてボタン（コマンド）を押すと、毎回同じ品質の飲み物（成果物）が出てきます。Skillsはまさにこれと同じです。「テーマと期間」を入れて「/sns-calendar」を押すと、毎回同じ品質のCSVが出てきます。

#### スライド5: どんなものがSkillsに向いているか
Skillsに向いている作業と向いていない作業があります。

向いているのは毎月・毎週繰り返す作業、手順が決まっている作業、複数のステップがある作業です。SNSカレンダー、週次レポート、議事録まとめはすべてSkillsに向いています。

向いていないのは1回だけやる作業、毎回内容が大きく変わる作業、手順が決まっていない作業です。

判断の目安は「来月も同じ手順でやるか？」です。答えがYesならSkills化を検討してください。

#### スライド6: Skillsの保存場所
Skillsは2種類の保存場所があります。

1つ目はプロジェクトスキルです。`.agents/skills/スキル名/SKILL.md` という場所に置きます。そのフォルダを開いているときだけ使えます。

2つ目は個人スキルです。ホームディレクトリの `~/.agents/skills/スキル名/SKILL.md` に置きます。どのプロジェクトでも使えます。

最初はプロジェクトスキルから始めましょう。自分のプロジェクトの中だけで使えればまずは十分です。

まとめます。Skillsは手順をコマンドにまとめる仕組みです。繰り返し作業に向いています。プロジェクトスキルから始めます。

#### スライド5: frontmatter（先頭の---の部分）
先頭の`---`で囲まれた部分をfrontmatterといいます。ファイルの「名札」の部分です。

`name` はコマンドの名前です。ここに `sns-calendar` と書くと、`/sns-calendar` で呼び出せるようになります。

`description` はこのSkillが何をするかの説明です。1行で書きます。

注意点があります。`name` に使える文字は英語の小文字とハイフンだけです。スペースや日本語は使えません。`name` を省略するとフォルダ名がそのまま使われます。

## レクチャー 6-4: Skillsの応用方法（想定時間: 7分）

### 画面構成
- 使用画面: スライド（メイン）→ Google AntiGravity画面（編集デモ）→ スライド
- 切替タイミング: バージョン管理の説明でSKILL.mdの編集画面を表示

### 読み上げ原稿

#### スライド1: 6-4レクチャータイトル
Skillsを作ったら終わりではありません。使いながら改善して、必要なら人に渡せる形にします。このレクチャーでは改善・共有・引き継ぎの方法を学びます。

#### スライド2: このレクチャーで達成すること
1つ目はSkillsのズレた部分を直す方法を理解すること。2つ目は他の人に渡すときの注意点を理解すること。3つ目は改善ログの考え方を身につけることです。

#### スライド3: Skillsがズレたときの直し方
Skillsを使っていると、「出力がちょっと違う」と感じることがあります。ターゲット設定が変わった、CSVの列を追加したい、新しいNG表現が出てきた、そういうときです。

修正の手順はシンプルです。

> **[画面切替: Google AntiGravity画面]**

まず SKILL.md を開きます。そして変わった部分だけを書き直します。

```
## チェック
- 重複ネタがないか
- 禁止表現がないか（追加: 「簡単に」「すぐに」もNG）  ← ここだけ変更
- 日付と曜日が一致しているか
```

最後に version 番号を上げます。

```
version: "1.1.0"  ← 1.0.0 から変更
```

> **[画面切替: スライド]**

バージョン番号の意味を簡単に説明します。

一番右の数字はパッチ。誤字修正や小さな調整です。1.0.0 → 1.0.1になります。

真ん中の数字はマイナー。手順の一部変更です。1.0.0 → 1.1.0になります。

一番左の数字はメジャー。大幅な作り直しです。1.0.0 → 2.0.0になります。

完璧に覚えなくてOKです。「変えたら番号を上げる」という習慣だけ作れれば十分です。

#### スライド4: 他の人に渡すときの注意点
Skillsは「仕組みの引き継ぎ」ができるのが強みです。

渡すものは3つです。`.agents/skills/sns-calendar/` フォルダ（SKILL.md本体）、GEMINI.mdのSNS設定部分（コンテキスト）、入力サンプル（実際に使った入力例）です。

伝えておくべきことも3つです。何のためのSkillか、毎月変える変数はどれか（テーマと期間だけです）、動かない場合はどこを見るかです。

「スキルを渡すときは使い方メモを一緒に渡す」これを覚えておいてください。

#### スライド5: 改善ログの考え方
改善ログを書いておくことで、「なぜこう書いたか」が3ヶ月後も分かります。チームで使うときに「なぜ変えたか」を共有できます。問題が起きたときに「どこが変わったか」を特定できます。

難しく考えなくていいです。最小限の1行でOKです。

```
## 変更ログ
- v1.1.0（2026-05-01）: ハッシュタグ上限を変更（以前は多すぎた）
- v1.0.0（2026-04-01）: 初版
```

完璧なログより、あるログを目指してください。1行でいいので書く習慣を作りましょう。

#### スライド6: セクション6まとめ + 次のセクションへ
セクション6全体を振り返ります。

Skillsは SKILL.md にコマンドの設計書を書く仕組みです。4要素は入力・出力・手順・チェックです。`/sns-calendar` でコマンド1つで動きます。ズレたら直してversionを上げます。他の人に渡すときは使い方メモを一緒に渡します。

次のセクションからはブログ記事の下書き量産に入ります。SNSより長い文章を、同じ分担方式で作っていきます。

### このレクチャーのポイント
- バージョン番号の説明は丁寧に。ソフトウェアに馴染みのない受講者には「更新のたびに番号を変えるルール」と伝える
- 「他の人に渡せる」というメリットはチーム利用を想定している受講者に刺さる
- 改善ログは「書かなくていい」と思わせないよう「1行でOK」と繰り返す
- つまずきやすい点: versionを変え忘れる受講者が多い。「変えたら番号を上げる」をセットで習慣化するよう促す
