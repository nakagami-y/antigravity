### 読み上げ原稿

#### スライド1: セクション7タイトル
セクション7は「ブログ下書き」です。このセクションでは、設計・執筆・編集・確認の4役でブログ記事を作ります。完成すると、テーマを渡すだけで1,500〜2,000字の下書きが20分で出てくるようになります。キーワードは「4人チーム」です。

「ブログ書かなきゃ…」と思いながら3ヶ月。白紙のエディタを開いては閉じる。この「真っ白な画面恐怖症」、経験ありませんか？ このセクションでは、4人のAIチームがその恐怖を解消します。

セクション4でSNSカレンダーを作り、5Cメソッドの全体像を体験しました。セクション7では「役割分担（Chain）」を深掘りします。1人で全部考えようとすると詰まりますが、作業を分けると速くなります。

#### スライド2: 7-1レクチャータイトル
ブログを書き始める前に「どうなっていれば完成か」を決めます。これがないと、出力を見ても「良いか悪いか」が判断できません。

#### スライド3: このレクチャーで達成すること
1つ目は「完成」の条件を先に書くこと。2つ目は見出し数・文字数・引用ルールを決めること。3つ目はNG表現を先にリストアップすることです。

#### スライド4: 合格ラインの決め方（3つだけ）
先に決めることは3つだけです。

1つ目は見出しの数です。例えば「H2が3〜5個」と決めます。

2つ目は文字数の範囲です。例えば「1,500〜2,000字」と決めます。短すぎず長すぎず、読まれる量を設定します。

3つ目は引用・NG表現のルールです。例えば「断定表現（絶対に、必ず）は禁止」と決めます。根拠のない断言はブログの信頼を下げます。

ゴールが決まれば、チェックが格段に楽になります。

```
## ブログ記事のルール
- 見出し: H2が3〜5個
- 文字数: 1,500〜2,000字
- 断定表現（絶対に、必ず）は使わない
- 数字には出典URLを必ずつける
```

期待される出力:

## レクチャー 7-2: 分担して作る（想定時間: 12分）

### 画面構成
- 使用画面: スライド → Google AntiGravity画面（ハンズオン 4ステップ）→ スライド
- 切替タイミング: 4ステップのハンズオンでGoogle AntiGravity画面に切り替え。各ステップ完了後に確認を挟む

### 読み上げ原稿

#### スライド1: 7-2レクチャータイトル
このレクチャーがセクション7の核心です。「役割分担（Chain）」の深掘りをします。

「1人で全部やらない」という考え方です。作業ごとに「担当」を変えることで、一気に書こうとしたときよりずっと速く、質の高い記事が書けます。

#### スライド2: このレクチャーで達成すること
1つ目は4つの役割と順番を理解すること。2つ目は各ステップを実際に体験すること。3つ目は「ステップごとに確認する」習慣を身につけることです。

#### スライド3: 4つの役割と順番（表）
4つの役割を確認します。

1番がStrategist（設計）。テーマ・構成・見出しを決める役割です。

2番がWriter（執筆）。本文を書く役割です。

3番がEditor（編集）。冗長な表現を削って表記を統一する役割です。

4番がQA（確認）。事実確認と引用チェックをする役割です。

この4つをすべてGoogle AntiGravityにやらせます。ただし各ステップで人間が「OK」を出してから次へ進みます。「全自動で任せる」のではなく「一緒に進む」イメージです。

#### スライド4: 各ステップの入出力（前半）
Step 1のStrategist（設計）は、テーマと対象読者と文字数目標を入れると、見出しリストが出てきます。

Step 2のWriter（執筆）は、Step 1で出た見出しリストを入れると、本文の下書きが出てきます。

各ステップの出力が次ステップの入力になります。パイプライン（流れ作業）のイメージです。

#### スライド5: 各ステップの入出力（後半）
Step 3のEditor（編集）は、本文の下書きを入れると、冗長表現を削った版と表記ゆれ修正済みの版が出てきます。

Step 4のQA（確認）は、編集済み本文を入れると、事実確認チェックリストと要修正箇所リストが出てきます。

QAは特に重要です。「AIが自分で書いたものを自分でチェックするのは弱い」ので、このステップの結果だけは人間が目を通すことを強くお勧めします。

#### スライド6: ハンズオン Step 1 — Strategist（設計）

> **[画面切替: Google AntiGravity画面]**

では実際に4ステップをやってみましょう。

Step 1は設計です。次のプロンプトを入力します。

```
テーマ：「在宅ワークの集中力を上げる方法」
対象：会社員（30代・在宅2年目）
文字数：1,500〜2,000字
H2見出しを3〜5個、H3を各2〜3個で構成案を作ってください
```

期待される出力:
```
## 構成案

### H2: 在宅ワークで集中力が下がる3つの原因
- H3: 自宅の誘惑（スマホ・テレビ）
- H3: 仕事モードへの切り替えが難しい

### H2: 今日から使える集中力アップ5つの方法
- H3: 作業前の「儀式」を作る
- H3: 90分+15分の集中サイクル
- H3: 視界の整理（デスク環境）

### H2: 1週間の実践スケジュール
- H3: 朝の準備ルーティン
- H3: 在宅ワーク中の時間管理
```

この構成を確認してください。「これで行こう」と思ったらStep 2に進みます。もし修正したい見出しがあればここで直します。

Step 2に進む前にここで止めて確認することが大事です。

#### スライド7: ハンズオン Step 2 — Writer（執筆）

では Step 2 です。先ほどの構成案を使って本文を書いてもらいます。

```
この構成案で本文を書いてください。
文字数は1,500〜2,000字、ですます調で
数字や主張には出典またはその旨の注記を入れてください
```

期待される出力:
```
# 在宅ワークの集中力を上げる方法

## 在宅ワークで集中力が下がる3つの原因

在宅ワークが浸透して数年が経ちますが、「オフィスより集中できない」

```

本文が出てきました。ざっと読んで「構成通りに書かれているか」を確認してください。文章の細かさはStep 3で直せます。ここでは構成の確認だけで大丈夫です。

#### スライド8: ハンズオン Step 3 — Editor（編集）

Step 3 は編集です。

```
この記事を編集してください。
以下の3点を確認・修正してください：
1. 冗長な表現（同じことを違う言い方で繰り返している箇所）
2. 表記ゆれ（「ユーザー」と「ユーザ」が混在するなど）
3. 「さらに」「また」「加えて」の連打（1段落1回以下に）
```

期待される出力:
```
【編集結果】

修正箇所:
1. 冗長表現3箇所を削除しました（元2,150字 → 1,850字）
2. 表記ゆれ: 「スマートフォン」「スマホ」を「スマートフォン」に統一
3. 「さらに」の連打を2箇所修正しました

編集済み本文:
...（修正済みの本文）
```

修正内容を確認してください。「意図と違う修正をされた」と感じたら元に戻すよう伝えられます。

#### スライド9: ハンズオン Step 4 — QA（確認）

Step 4 は確認です。

```
この記事の品質チェックをしてください。
確認してほしいのは：
1. 数字・統計に出典はあるか（ない場合は箇所を教えて）
2. 引用の書き方は正しいか
3. 断定表現（絶対に・必ず）が使われていないか
```

期待される出力:
```
【品質チェック結果】

要確認箇所:
1. 「90分の集中サイクル」→ 出典が明記されていません
   → 「研究によると」ではなく具体的な出典を記載することを推奨
2. 引用表記: 適切です
3. 断定表現: 「必ず試してください」が1箇所 → 修正推奨

修正不要な箇所:
- 見出し構成: H2×3個、各H2にH3が2〜3個。問題なし
- 文字数: 1,873字。目標範囲内
```

> **[画面切替: スライド]**

この結果を見て、「90分の集中サイクル」の出典を追加するかどうかを判断します。これは人間が決めることです。

#### スライド10: まとめ + 次へ
まとめます。設計 → 執筆 → 編集 → 確認の順に進めます。一気にやろうとしません。分けることがコツです。各ステップの出力を確認してから次へ進みます。

次のレクチャーでは最低限のチェックポイントを確認します。

### このレクチャーのポイント
- 4ステップのハンズオンは実際に画面で動かして見せる。受講者が後で再現できる形にする
- 各ステップの間に「ここで確認」を声で案内する。受講者がリズムをつかめる
- Step 4（QA）の結果は必ず人間が確認することを強調する
- 撮影時の注意: 4ステップで約12分。1ステップ約3分のペースで進める
- つまずきやすい点: Step 2の本文が長くなりすぎることがある。「2,000字以内」の指示を明示することで防げる

### 読み上げ原稿

#### スライド1: 7-3レクチャータイトル
「全部チェックしなきゃ」と思うと挫折します。最低限この3つだけで大丈夫です。一次情報の確認、引用の書き方、表記ゆれの3点です。

#### スライド2: このレクチャーで達成すること
1つ目は最低限のチェック3点を理解すること。2つ目は実際にGoogle AntiGravityでチェックを依頼すること。3つ目は結果の読み方を覚えることです。

#### スライド3: チェック観点リスト（3点）
3つのチェックポイントを説明します。

1つ目は一次情報チェックです。数字（統計・調査結果）に出典はあるか、固有名詞（会社名・商品名）は正確かを確認します。AIが書いた文章には、もっともらしいが実際には根拠のない数字が入ることがあります。ここは必ず確認してください。

2つ目は引用チェックです。引用元が明記されているか、引用文言が正確かを確認します。

3つ目は表記ゆれチェックです。「AI」と「A.I.」が混在していないか、「ユーザー」と「ユーザ」など送り仮名のゆれがないかを確認します。

#### スライド4: ハンズオン（チェック実施）

> **[画面切替: Google AntiGravity画面]**

では実際にチェックを依頼してみましょう。

```

1. 一次情報チェック: 数字や統計に出典はあるか。固有名詞は正確か
2. 引用チェック: 引用元が明記されているか、引用文言が正確か
3. 表記ゆれチェック: 同じ言葉が違う書き方になっていないか
```

```
2. 引用チェック

```

この結果の読み方を説明します。「要確認」と出た箇所は自分で調べるか、記述を「〜といわれています」のような表現に変えましょう。「問題なし」はそのままで大丈夫です。表記ゆれは統一を選んでください。

#### スライド3: 毎回変えるもの vs 変えないもの
ポイントは「何を固定して何を変えるか」です。

毎回変えるものはテーマ、対象読者、参考URL（任意）です。テーマが違えばこの3つだけ変えます。

毎回変えないものは文字数、見出し構成、NG表現、出典ルールです。これを SKILL.md に一度書いておけば、毎回指示しなくて済みます。

#### スライド4: SKILL.md の作成（ハンズオン）

> **[画面切替: Google AntiGravity画面]**

```
ブログ下書きを作るSkillのSKILL.mdを作ってください
ファイルパスは .agents/skills/blog-draft/SKILL.md

毎回変えるもの:
- テーマ
- 対象読者
- 参考URL（任意）

固定ルール（毎回変えないもの）:
- 文字数：1,500〜2,000字
- H2：3〜5個、各H2にH3を2個
- NG表現：断定表現（絶対・必ず）は禁止
- 出典：数字には必ず出典または注記を記載
```

```

- 必須: テーマ
- 必須: 対象読者
- 任意: 参考URL

- 文字数：1,500〜2,000字
- H2：3〜5個、各H2にH3を2個
- NG表現：断定表現（絶対・必ず）は禁止

## 手順
1. Strategist: 見出し構成を作る
```
