# わどのコンテンツ戦略

> **ひかりのコンテンツ戦略**: `../hikari/content-strategy.md` を参照。わどのマスターコンテンツがひかりのYouTube台本の素材にもなる。

## マスターコンテンツループ（全体像）

わどのコンテンツ制作は「作って終わり」ではなく、循環するループで回す。

```
【ネタ収集】毎日・随時
  面談メモ / 受講生の声 / 作業ログ / AI最新 / 競合 / SNS反応 / 書籍
    |
【週次棚卸し】月曜30分
  キャプチャ見返し --> テーマ2-3個選ぶ --> マスター昇格判定
    |
【マスター制作】週2-3本
  大マスター: note記事 / YouTube台本 / セミナー台本
  中マスター: メルマガ / Q&A回答 / 事例レポート
    |
【チャネル展開】火〜金
  集客系: X / IG / note / YouTube / ショート動画
  ファネル系: メルマガ / LINE / オプチャ --> セミナー誘導
    |
【反応回収】随時
  伸びた投稿 / コメント / 質問 --> 次のネタとしてキャプチャ
    |
  ネタ収集に戻る（ループ）
```

---

## ネタの7ソース

| # | ソース | 具体例 | 頻度 |
|---|--------|--------|------|
| 1 | 面談・個別相談 | 「今日こういう質問が来た」「この悩みが多い」 | 毎回 |
| 2 | 受講生の声 | 成果報告、つまずき、DM、コミュニティ投稿 | 随時 |
| 3 | 自分の作業・実験 | 「AIでこれ試した」「こうやったら上手くいった」 | 毎日 |
| 4 | AI/テック最新情報 | 新ツール、アップデート、業界ニュース | 毎日 |
| 5 | 競合の動き | 競合の記事、セミナー、商品、SNS投稿 | 週1 |
| 6 | SNS反応 | 伸びた投稿、コメントで多い反応、質問 | 毎日 |
| 7 | 書籍・記事・動画 | 読んだ本、参考フレームワーク、動画の学び | 随時 |

**一番強いネタは1と2**。面談での生の質問と受講生のリアルな声は競合が持っていない一次情報。AI臭もゼロ。ここの取りこぼしが最大の損失。

### キャプチャの鉄則

- 30秒で書ける粒度でOK。完璧なメモじゃなくていい。キーワードと文脈だけ残す
- 1キャプチャ = 1ノート（混ぜない）

## 3ティア定義（素材 --> マスター --> 展開の階層）

### Tier 1: 大マスター（Big Master）

| 項目 | 内容 |
|------|------|
| 頻度 | 週1-2本 |
| 形式 | note記事（3,000-5,000字+）、YouTube台本（10-15分）、セミナー台本（30-90分） |
| 特徴 | 深い一次情報、体系的な構成、独自の切り口 |
| 展開力 | 10+チャネルに展開可能 |
| 作成タイミング | 月曜日（週の最初に集中して作成） |

### Tier 2: 中マスター（Medium Master）

| 項目 | 内容 |
|------|------|
| 頻度 | 週3-5本 |
| 形式 | メルマガ1号（800-1,500字）、相談メモ要約、受講生成果レポート、Q&A回答 |
| 特徴 | 大マスターより軽量、特定の切り口に絞った内容 |
| 展開力 | 3-5チャネルに展開可能 |
| 作成タイミング | 木曜日 |

### Tier 3: 素材（Raw Material）

| 項目 | 内容 |
|------|------|
| 頻度 | 毎日のキャプチャ |
| 形式 | 作業ログ、音声メモ書き起こし、スクリーンショット、データ、失敗メモ、受講生の声、相談メモ |
| 特徴 | 未加工の一次情報。マスターコンテンツの原料 |
| 収集 | 日常的にキャプチャ。月曜に棚卸し |

## ティア昇格判定ルール

素材が以下の3つ以上に該当する場合、大マスターの原料として優先する。

| # | 判定基準 | 説明 |
|---|---------|------|
| 1 | 一次体験がある | 自分で実際にやった |
| 2 | 数字/データがある | Before --> Afterが示せる |
| 3 | 受講生/ユーザーの声がある | 社会的証明として使える |
| 4 | 競合との差別化ポイントがある | 他の人が言っていない切り口 |
| 5 | AI活用3本柱のいずれかに直結する | 商品数UP / 収益機会UP / サポート自動化 |
| 6 | タイムリーなトレンドと接続できる | 今このタイミングだから刺さる |

2つ以下の場合は中マスターの素材として検討するか、追加情報が集まるまでストック。

## 配信マトリクス（マスター --> チャネル変換ルール）

1本のマスターコンテンツからどれだけの展開ができるかの目安。

| マスターの形式 | X投稿 | ショート動画 | LINE | メール | note | OC |
|--------------|:-----:|:----------:|:----:|:-----:|:----:|:--:|
| note記事 | x10 | x3 | x1 | x1 | - | x6 |
| YouTube台本 | x5 | x3(切り抜き) | x1 | x1 | x1(要約) | - |
| セミナー台本 | x5 | x3 | x1 | x1 | x1(レポート) | x6 |
| メルマガ | x3 | - | x1 | - | - | - |
| 相談メモ要約 | x1 | - | - | x1 | - | - |
| 受講生成果 | x2 | - | - | x1 | - | - |

### マスターから抽出すべき6要素

1. **コアメッセージ** -- 1文で要約できる核心
2. **キーポイント** -- 3-5個の柱（各1つがX投稿1本になる）
3. **ストーリー/体験談** -- Before --> After（ショート動画・LINE向き）
4. **データ/数字** -- 具体的な証拠（フック候補・説得力の根拠）
5. **引用可能な一言** -- そのままフックに使える強い文
6. **今日の一手** -- 読者がすぐやれる1アクション

### X（旧Twitter）

| 項目 | 内容 |
|------|------|
| 役割 | 刺す（信頼構築）。診断/反論潰し/フレーム/具体で証明 |

| 項目 | 内容 |

| 変換ルール | マスターとは別切り口で記事化（コピペではない） |
| 成約ルートでの位置 | ステップ2「信頼構築」-- 一貫性と専門性 |

### LINE（公式）

| 項目 | 内容 |
|------|------|
| 役割 | 分岐（中枢）。配布 + 宿題 + 期限 + セグメント |
| 温度 | 中高。友人と1対1でカフェで話す |
| 性質 | 1対1の個室（DM感） |
| 変換ルール | コアメッセージ + 1つの具体例 + CTA |
| 成約ルートでの位置 | ステップ2-3「信頼構築 --> 問題の自覚」 |
| 特徴 | 刺さるメッセージをターゲット絞りで配信。次の1ステップに送ることに集中 |

### YouTube

| 項目 | 内容 |
|------|------|
| 役割 | 資産（体系化）。設計 --> 事例 --> 手順 --> 宿題 |
| 温度 | 中〜高（長尺は構造的、ショートはテンション高め） |
| 変換ルール | 大マスターの体系化。冒頭30秒が勝負 |
| 成約ルートでの位置 | ステップ1-2「日常接触 --> 信頼構築」 |

### LINEオープンチャット

| 項目 | 内容 |
|------|------|
| 役割 | 定着（提出）。週1チャレンジ + 提出 |
| 性質 | みんなが見える場所（公開・半公開）。熱狂的・盛り上がり重視 |
| 変換ルール | コアメッセージは固定、フック/具体例/トーンはOC別に最適化 |
| 成約ルートでの位置 | ステップ1「日常接触」-- 見える化で熱量とコミュニティを可視化 |
| 注意 | 同一テーマでもOCごとにメッセージを最適化する。コピペ配信は禁止 |

#### OC一覧と最適化ポイント

| OC | ターゲット像 | 刺すポイント |
|----|------------|------------|
| ライティング | ブログ・note・SNS・メルマガなど文章を書く人 | 制作時間の短縮（Before --> After数字） |
| 画像生成 | Midjourney・DALL-E・SD等で画像を作る人 | 生成以外の周辺作業（ファイル管理・キャプション等） |
| 開発 | エンジニア・プログラマー | 技術的な差別化ポイント |
| 漫画 | 漫画制作者 | 「描く以外」の雑務をAIに任せる |
| マネタイズ | 収益化・コンテンツ販売に興味がある人 | 打席数 = 売上。制作速度がボトルネック |
| AI初心者 | AIをほとんど使ったことがない人 | 専門用語ゼロ。日常の願望から入る |

#### OC別カスタマイズする要素

| 要素 | やること |
|------|---------|
| フック | そのOCメンバーの「日常の痛み」に直接刺す1行目にする |
| 具体例 | そのOCメンバーが普段やっている作業に置き換える |
| 専門用語の量 | 開発 --> 多めOK / AI初心者 --> ゼロ / 他 --> 最小限 |
| トーン | マネタイズ --> ビジネス寄り / AI初心者 --> 親しみやすさ重視 / 開発 --> 対等な技術トーク |

## 成約に至る6ステップ

成約に至るルートは1本だけ。各チャネルの仕事は「次の1ステップに送ること」だけに集中する。

| ステップ | 心理 | 主なチャネル | 役割 |
|----------|------|--------------|------|
| 1. 日常接触 | 「この人、役に立つな」 | オプチャ / X | 存在感・有用性 |
| 2. 信頼構築 | 「この人の言うこと、信じられるな」 | メルマガ / note | 一貫性・専門性 |
| 3. 問題の自覚 | 「自分、このままだとヤバいかも」 | メルマガ / LINE | 課題の言語化 |
| 4. 解決策の提示 | 「それを解決するセミナーがあるのか」 | セミナー告知 | オファー認知 |
| 5. 体験 | 「実際に体験して、やり方が見えた」 | セミナー / ウェビナー | 体験・納得 |
| 6. 決断 | 「自分の場合どうすればいいか相談したい」 | 個別相談 | クロージング |

## フック設計フレームワーク

フックは「驚かせる言葉」ではなく、特定の心理メカニズムを起動して次の1ステップを踏ませる装置。感覚ではなく構造で設計する。

### 5つの心理メカニズム

| # | メカニズム | 原理 | フックでの使い方 |
|---|-----------|------|----------------|
| 1 | 好奇心ギャップ | 「知識の欠け」を自覚 --> 埋めたい衝動 | 重要だが未完了の情報を提示し、即答しない |
| 2 | 損失回避 | 損失は同程度の利得より心理的インパクト大 | 「避けたい損失」「やると損する落とし穴」を冒頭に |
| 3 | 未完了効果 | 中断・未完了タスクは記憶に残りやすい | 結論の一部を見せて未完了状態を作る |
| 4 | 社会的証明 | 他者の判断・多数派に引っ張られる | 「X人がやっている」「事例の数字」を冒頭に |
| 5 | 処理流暢性 | 処理しやすい刺激を好意的に評価 | 短い・具体的・平易・数字明確・動詞で動く |

### 8型フレームワーク（H1〜H8）

| 型ID | 名称 | 構造（骨格） | 主心理トリガー |
|------|------|------------|-------------|
| H1 | ベネフィット先出し | {対象}は{方法}で{成果}できます | 処理流暢性 |
| H2 | 痛み --> 回避 | {よくある失敗}、それ{原因}です | 損失回避 |
| H3 | 好奇心ギャップ | {問い}の答え、{よくある誤解}です（最後に答え） | 情報ギャップ |
| H4 | 逆張り（反常識） | {常識}は{NG}、{理由}は{一言} | 注意喚起 |
| H5 | 数字で具体化 | {時間/回数}で{成果}する{手順数} | 処理流暢性 + 具体 |
| H6 | 証拠・事例 | {実例}で{結果}、やったことは{施策3つ} | 社会的証明 |
| H7 | 参加型（質問/診断） | {質問}。当てはまったら{次の一言} | 自己関与 |
| H8 | 期限・希少 | {期限}までに{行動}しないと{損} | 損失回避 + 希少性 |

### フック設計の絶対基準

1. **1行目で心理メカニズムを起動する** -- どれを狙うか決めてから書く
2. **フックなしの投稿・配信は禁止** -- 「お知らせです」「ご案内です」から始まる文は即却下
3. **フックは最低3案出して最強を選ぶ** -- H1〜H8の異なる型で3案書いて比較
4. **約束と回収を一致させる** -- フックで作った期待を本文で必ず回収する。回収しない = 釣り = 信頼崩壊

## 媒体別フック最適設計

| 媒体 | フックが勝負する場所 | 相性の良い型 | 主KPI |
|------|-------------------|-----------|------|
| X | 冒頭1行目の13文字 | H4, H5, H2, H7 | エンゲージメント --> 遷移/クリック |
| note記事 | タイトルが9割 | H5, H4, H3 | クリック --> 読了 --> CTA |
| LINE | 通知プレビュー20文字 | H1, H7, H8 | 開封率 --> クリック率 |
| 短尺動画 | 0〜3秒（映像 + 字幕） | H1/H6（先出し）, H3/H7（問い） | 3秒後残存 --> 完了率 |
| YouTube長尺 | タイトル/サムネ --> 冒頭30秒 | H1/H6, H2, H3 | CTR --> Intro（30秒後残存） |
| 告知文 | 冒頭1行 | H2, H8 | 開封/クリック --> 申込 |

### 媒体別フックサンプル（代表例）

#### X

- その努力、実は逆効果。成果が出ない原因は「順番」です
- 3分で終わるのに、毎回30分かかる人の共通点
- 伸びない投稿の9割、冒頭1行目で損してる
- 保存用: 今日から使える冒頭テンプレ8種

#### LINE

- 【3分】文章の冒頭だけ直すテンプレ、要りますか?
- 質問: あなたの文章、最初に"得"が書けてます?
- 今の投稿、冒頭で損してるか2問で判定します

#### note記事タイトル

- フックの作り方: 冒頭1行で離脱を止める8つの型
- 3分でできる文章改善: 読まれる書き出しテンプレ集

## 横展開SOP（チャネル変換の原則）

マスターコンテンツを各チャネルに変換する際のルール。

| チャネル | 変換ルール |
|---------|----------|
| X投稿 | キーポイント1つ = 投稿1本。フックは各投稿で独立設計（マスターのフックをコピペしない） |
| ショート動画 | ストーリー1つ or 手順1つ = 台本1本。フックは映像 + 字幕用に再設計 |
| LINE | コアメッセージ + 1つの具体例 + CTA。DM感を維持 |
| メール | コアメッセージ + Why + 今日の一手 + CTA。件名はH型で独立設計 |
| note（転用） | マスターとは別切り口で記事化。同じ内容のコピペではない |
| OC | コアメッセージは固定、フック/具体例/トーンはOC別に最適化 |

### 横展開の注意点

- **コピペ厳禁**: 同じ文章をそのまま別チャネルに貼らない。各チャネルの文法に合わせて再設計する
- **フックは毎回独立設計**: マスターのフックをそのまま使うのではなく、チャネル特性に合わせて再設計する
- **CTA先はレーン統一**: 同一コンテンツ内で0 --> 10と10 --> 100のCTAを混ぜない

### 月曜の棚卸し手順（30分）

6. 使う素材をまとめ、マスター執筆に入る

## 反応回収ルール

コンテンツは「作って終わり」ではない。反応がネタに還流するループを回す。

### 自動回収

| 反応の種類 | 次のアクション |
|-----------|--------------|
| X投稿の高エンゲージメント | 深掘りnote記事 or セミナーネタ候補としてキャプチャ |
| コメントでの質問 | Q&A型の中マスター候補 |
| LINE返信の声 | 受講生の声として蓄積 |

### 月次フィードバック

| 分析 | 反映先 |
|------|--------|
| 効いたテーマの特定 | 翌月のテーマ選定に優先反映 |
| 効かなかったテーマ | 切り口を変えるか、捨てるか判断 |
| ベンチマーク比較 | 数字の変化をキャプチャ --> 事例ネタに |

## A/Bテスト設計（フック改善サイクル）

### 基本原則

- 目的KPIは「1つ」に絞る
- 変えるのは「フックだけ」に近づける
- 勝ち要素を積み上げる

### 媒体別KPIと判定

| 媒体 | 一次KPI | A/Bで変える要素 | 判定の読み方 |
|------|--------|---------------|------------|
| X | エンゲージメント | 1行目の型（H4 vs H5）、質問有無 | 反応高いが遷移低い --> 面白いが行動弱い |
| Instagram | 保存/シェア | 結論先出し vs 共感、数字有無 | 保存低い --> 「役立ち」が弱い |
| 短尺動画 | 初期残存（3秒後） | 0〜3秒の映像、字幕1行目 | 初期残存低い --> 0〜3秒が弱い |
| YouTube長尺 | CTR + Intro | タイトル/サムネ + 冒頭 | CTR高 + Intro低 = 期待不一致 |
| LINE | 開封率 --> クリック率 | 1行目（提案 vs 質問）、期限有無 | 開封低い --> 件名。クリック低い --> オファー |
| ブログ | 検索CTR | 見出し（数字、対象明確化） | CTR高 + 直帰高 = 見出しと本文不一致 |

### 改善ループ

1. 設計: 目的KPI固定 --> 仮説1つ --> A/B案作成
2. 実行: 期間・曜日・時間帯を揃えて配信
3. 分析: ファネル分解で診断（クリック弱い? 残存弱い?）
4. 展開: 勝ち型をテンプレート化 --> ライブラリ更新
